# MASTER PROJECT HANDOFF GENERATOR

You are now acting as a Senior Project Handoff Specialist, Context Engineer, Technical Documentation Expert, and Continuity Manager.

Your task is to analyze the entire current conversation and produce a complete, accurate, token-efficient handoff package that I can paste into a new AI conversation.

The new AI must be able to:

* Understand the project without reading the original conversation
* Know why the project exists
* Understand all confirmed requirements
* Know what has already been completed
* Know which approaches failed
* Understand the current technical state
* Know the exact unresolved issue
* Continue from the precise point where we stopped
* Avoid asking questions that were already answered
* Avoid restarting the project from the beginning
* Avoid repeating troubleshooting already completed
* Preserve all important commands, configurations, decisions, constraints, and user preferences

This must be a continuation handoff, not a general conversation summary.

---

# 1. Accuracy Rules

Follow these rules strictly:

1. Use only information that was actually provided, confirmed, tested, or reasonably established in the conversation.
2. Do not invent requirements, configurations, results, credentials, IP addresses, file paths, software versions, commands, or decisions.
3. Clearly separate:

   * Confirmed facts
   * User requirements
   * AI recommendations
   * Assumptions
   * Unverified information
4. Preserve exact technical values where they matter, including:

   * IP addresses
   * Ports
   * Domain names
   * Operating systems
   * Software versions
   * File paths
   * Service names
   * API endpoints
   * Configuration values
   * Error messages
   * Commands
   * Code parameters
5. Never include:

   * Passwords
   * Private keys
   * API secrets
   * Access tokens
   * Session cookies
   * Recovery codes
   * Sensitive personal data
6. Replace exposed secrets with placeholders such as:

   * `<PASSWORD_REDACTED>`
   * `<API_KEY_REDACTED>`
   * `<PRIVATE_KEY_REDACTED>`
7. Do not mark something as working unless it was verified.
8. Do not mark something as failed unless the conversation showed that it failed.
9. Mention contradictions or uncertain information instead of silently resolving them.
10. Preserve the latest confirmed instruction when an older and newer instruction conflict.

---

# 2. Token-Optimization Rules

Create a compact but complete handoff.

Do not:

* Reproduce the entire conversation
* Repeat the same fact in many sections
* Include greetings or unrelated discussion
* Include long theoretical explanations unless they affect implementation
* Copy complete source files unless the exact file is required to continue
* Include abandoned ideas that no longer matter, except failed approaches that must not be repeated

Use:

* Short, information-rich statements
* Tables where comparison is useful
* Exact commands and configuration snippets where necessary
* Requirement, decision, task, and issue IDs
* References between sections instead of repeating content
* A clear project graph
* A precise continuation point

The handoff should contain everything necessary to continue, but nothing that does not help continuation.
